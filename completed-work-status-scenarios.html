CompletedWork Status Logic

CompletedWorkUpdater._update_status — before vs after PR #987  ·  auto_approve_payments = True

What Changed

Three behavioural changes in this PR

All changes are in CompletedWorkUpdater._update_status and the new helper _is_completed_work_approved. The legacy standalone function _update_status_set_saved_fields_and_get_payment_accrued is unchanged.

Breaking
Optional deliver units now block approval
Previously, optional deliver units were completely ignored when deciding whether a completed work is approved. Now, if a payment unit has any optional deliver units, at least one must have an approved visit for the completed work to be approved. For managed opps, the optional DU's visit must also have review_status = agree.
Fix
incomplete promoted to pending
The default model status is incomplete. Previously, a completed work could stay stuck as incomplete even after visits were submitted if not all required approvals were met. Now the else branch promotes incomplete → pending, correctly signalling that work has been done but not yet fully approved.
Clarified
Terminal statuses are now explicitly preserved
rejected, approved, and over_limit are not touched by the else branch. If re-evaluation fires on a CW in one of those states and neither the rejection nor approval conditions are met, the status is left as-is. Only incomplete is promoted.
Logic Diff

How the approval check was restructured

Both versions share the same rejection check in branch 1. The approval check (branch 2) and the fallback (branch 3) are what changed.

Before (acc6f168)
Branch 1 — rejection (same):
Any visit with status = rejected → CompletedWorkStatus.rejected

Branch 2 — else (all-or-nothing):
Calls _get_required_delivery_approvals(visits, cw)
Iterates only required deliver units. For each, queries DB for a visit where:
• status = approved
• (managed only) review_status = agree
If all required DUs have such a visit → approved
Optional deliver units: not checked at all

No else branch: status left unchanged if neither fires.
After (this PR)
Branch 1 — rejection (same):
Any visit with status = rejected → CompletedWorkStatus.rejected

Branch 2 — elif (per-DU counts, optionals included):
Calls _is_completed_work_approved(cw) using pre-computed per-DU visit counts:
• All required DUs: approved_count > 0 (managed: also agree_count > 0)
• If optional DUs exist: ≥1 optional must have approved_count > 0 (managed: also agree_count > 0)
→ approved

Branch 3 — elif (new fix):
If current status is incomplete → pending

Otherwise: status left unchanged (rejected/approved/over_limit/pending preserved).

Legacy function: _update_status_set_saved_fields_and_get_payment_accrued (NOT changed by this PR)

This standalone function still exists in the same file and is used by the populate_completed_work_saved_fields management command only. It uses a substantially different approach:

  • Excludes duplicate visits before evaluation: .exclude(status__in=[duplicate]). The new CompletedWorkUpdater does not.
  • No concept of required vs optional deliver units. It checks whether ALL visits (across all deliver units) are approved: elif all(status == approved for status, *_ in visits). A single non-approved visit blocks approval, regardless of which DU it belongs to.
  • Managed agreement is checked globally, not per-DU. After setting approved, it checks if all visits have review_status = agree. If not, it resets to pending. This is a two-step approach vs the new per-DU count approach.
  • No incomplete → pending transition. Same gap as the old CompletedWorkUpdater before this PR.
  • This function calls completed_work.save() directly (not bulk update), making it unsuitable for batch processing.
