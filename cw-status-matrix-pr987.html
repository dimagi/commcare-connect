Core change
Per-DU approval replaces global all-visits check
Old code: all(status == approved for status in visits) — every visit across all deliver units must be approved. New code: checks each required DU has ≥1 approved visit. Extra non-approved visits on the same DU no longer block approval.
Breaking
Optional DUs now block approval
Old code ignored optional DUs entirely. New code requires ≥1 optional DU to have an approved visit (via any()) for the CW to be approved.
Fix
incomplete promoted to pending
Old code had no fallback — CWs could stay stuck as incomplete forever even after visits were submitted. New code promotes incomplete → pending once visits exist.
Structural
Managed agreement check integrated into per-DU logic
Old code: two-step — set approved, then a separate if managed block demotes to pending. New code: _is_delivery_approved checks both approved > 0 and agree > 0 per DU in one pass. No demotion step.
Logic Diff

How _update_status was restructured

Both versions share branch 1 (rejection). Everything else changed.

main (8493ee05)
Branch 1 — rejection (same):
Any visit with status = rejected → rejected

Branch 2 — elif (global all-visits):
all(status == approved for status, *_ in visits)
Checks every visit on the CW, regardless of deliver unit. A single non-approved visit anywhere (e.g. over_limit, pending, duplicate) blocks approval.

Separate managed demotion (not elif):
if managed and not all(review_status == agree) and status == approved → pending
Runs after the approval check as a separate if. Can demote a just-set approved back to pending.

No fallback: If neither rejection nor all-approved fires, status is left unchanged.
PR #987
Branch 1 — rejection (same):
Any visit with status = rejected → rejected

Branch 2 — elif (per-DU counts):
_is_completed_work_approved(cw) using pre-computed per-DU visit counts:
• Each required DU: approved > 0 (managed: also agree > 0)
• If optional DUs exist: ≥1 optional must also pass
→ approved

Branch 3 — elif (new):
If current status is incomplete → pending

Otherwise: status left unchanged (rejected / approved / over_limit / pending preserved).

Edge Cases

Edge cases and subtleties

Edge Case Behaviour
over_limit visits Old: Poisons all(status == approved) — blocks approval even if another visit on the same DU is approved.
New: Not rejected (skip branch 1). Not approved (doesn't contribute to approved counts). Does increment unit_counts (feeds completed_count). Effectively neutral for status decisions in new code.
duplicate / trial visits Same as over_limit. Old code: blocks all(). New code: neutral. Neither code version excludes them from the visit queryset (unlike the legacy standalone function which calls .exclude(status__in=[duplicate])).
No required deliver units (vacuous approval) Old: If no non-approved visits exist in queryset, all([]) = True → approved. If optional DU has visits, those visits must all be approved too (since global check).
New: all(... for x in []) = True → all_required_approved = True. Then checks optionals: any() must find ≥1 approved optional DU. If no optional DUs exist either, returns True.
Multiple visits on the same required DU Old: All visits must be approved — if DU has 1 approved + 1 pending, approval fails.
New: Only needs ≥1 approved per DU. Extra non-approved visits are ignored. This is the most impactful change. Confirmed by test: test_managed_opp_completed_work_updated_to_approved_with_same_unit_over_limit passes on PR, fails on main.
Managed agreement: global vs per-DU Old: Two-step. Sets approved, then checks all(review_status == agree) globally. One non-agree visit anywhere demotes to pending.
New: Integrated. Each DU must have agree > 0. If a DU has both an agree visit and a non-agree visit, the DU still passes (only needs ≥1). This means a non-agree visit on a DU with another agree visit no longer blocks approval.
Caller filter interactions bulk_approve_completed_work, bulk_update_payment_accrued, and the management command all .exclude(status=rejected). Only update_payment_accrued_for_user(incremental=False) passes all statuses — that's the only path where rejected CW re-evaluation matters.
Legacy Function
_update_status_set_saved_fields_and_get_payment_accrued (NOT changed by this PR)

This standalone function still exists in the same file and is used by the populate_completed_work_saved_fields management command only. It uses a substantially different approach:

  • Excludes duplicate visits before evaluation: .exclude(status__in=[duplicate]). Neither the old nor new CompletedWorkUpdater does this.
  • No concept of required vs optional deliver units. Checks whether ALL visits are approved: all(status == approved for ...). Same global approach as the old CompletedWorkUpdater.
  • Managed agreement is checked globally, not per-DU. After setting approved, checks if all visits have review_status = agree. If not, resets to pending. Same two-step approach as old code.
  • No incomplete → pending transition. Same gap as the old CompletedWorkUpdater.
  • Calls completed_work.save() directly (not bulk update), making it unsuitable for batch processing.
